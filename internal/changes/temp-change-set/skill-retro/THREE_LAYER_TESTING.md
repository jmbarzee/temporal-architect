# Three-Layer Testing for Temporal

Temporal code divides naturally into three testable layers. Each layer mocks only its immediate dependency — not anything deeper.

```
┌──────────────────────────────────────────┐
│  Workflows          mocks: Activities    │  fast, no Docker
├──────────────────────────────────────────┤
│  Activities         mocks: Client        │  fast, no Docker
├──────────────────────────────────────────┤
│  Clients            mocks: nothing       │  needs Docker
└──────────────────────────────────────────┘
```

| Layer | File | Mocks | Build tag | Speed |
|---|---|---|---|---|
| Workflows | `workflows_test.go` | Generated activities interface | *(none)* | fast |
| Activities | `activities_test.go` | Hand-written client interface | *(none)* | fast |
| Clients | `client_test.go` | Nothing (real system) | `integration` | slow |

---

## Why Three Layers?

**Workflows** test orchestration: are activities called in the right order, with the right inputs, and do failures propagate correctly? They cannot reveal bugs inside activities.

**Activities** test validation and error handling: does invalid input fail before touching the external system? Does a client error bubble up correctly? They cannot reveal bugs in the real client implementation.

**Client tests** test the real system: does the client actually speak the correct protocol, handle error codes correctly, and return the right types? Mocks can hide all of this.

Each layer catches bugs the others cannot.

## Mock Generation with mockery

Mocks are generated by [mockery](https://vektra.github.io/mockery/) from a `.mockery.yaml` config file. Install it once:

```bash
go install github.com/vektra/mockery/v2
```

Then run:

```bash
go generate -run mockery
```

or via Makefile: `make mocks`.

### `.mockery.yaml` — configuration

```yaml
with-expecter: true              # enables .EXPECT() style assertions
mockname: "Mock{{.InterfaceName}}"
outpkg: "{{.PackageName}}"
filename: "{{.InterfaceName | snakecase}}.go"
dir: "mocks/{{.InterfaceDirRelative}}"

resolve-type-alias: false
disable-version-string: true
issue-845-fix: true

packages:
  # ── Generated interfaces (from proto) ──────────────────────────────────────
  # Mock these in workflow tests: they represent activities your workflow calls.

  github.com/myorg/myproject/gen/myservice/v1:
    config:
      dir: "mocks/myservice/v1"
    interfaces:
      MyServiceActivities:

  # ── Implementation interfaces (hand-written clients) ───────────────────────
  # Mock these in activity tests: they represent the external system your
  # activity talks to.

  github.com/myorg/myproject/lib/myservice/v1:
    config:
      dir: "mocks/myservice/v1"
      outpkg: "myservicev1"
    interfaces:
      MyServiceClient:
```

Two categories of entries:
1. **Generated interfaces** — one entry per proto service; used in workflow tests
2. **Implementation interfaces** — one entry per client interface; used in activity tests

Add a new entry whenever you add a new service or a new client interface.

## Layer 1: Workflow Tests

Workflow tests use the Temporal test suite's `WorkflowTestSuite`. They mock the activities interface so the workflow can be driven to any outcome without touching real systems.

```go
// workflows_test.go
package myservice

import (
    "testing"

    pb "github.com/myorg/myproject/gen/myservice/v1"
    mocks "github.com/myorg/myproject/mocks/myservice/v1"
    "go.temporal.io/sdk/testsuite"
)

func TestDeployClusterWorkflow_Execute(t *testing.T) {
    tests := []struct {
        name        string
        clusterName string
        createErr   error
        wantErr     bool
    }{
        {
            name:        "success - cluster deployed",
            clusterName: "prod",
            wantErr:     false,
        },
        {
            name:        "error - activity fails",
            clusterName: "prod",
            createErr:   errors.New("quota exceeded"),
            wantErr:     true,
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            // Fresh environment for every test case
            env := (&testsuite.WorkflowTestSuite{}).NewTestWorkflowEnvironment()
            mockActivities := mocks.NewMockMyServiceActivities(t)

            // Register mock implementation
            env.RegisterActivity(mockActivities)

            // Set expectations: only for the happy-path-reachable path
            mockActivities.EXPECT().
                CreateThing(mock.Anything, mock.Anything).
                Return(&pb.CreateThingOutput{Id: "thing-1"}, tt.createErr)

            env.ExecuteWorkflow(func(ctx workflow.Context, input *pb.DeployClusterInput) error {
                wf, err := workflows.DeployCluster(ctx, &pb.DeployClusterWorkflowInput{Req: input})
                if err != nil {
                    return err
                }
                _, err = wf.Execute(ctx)
                return err
            }, &pb.DeployClusterInput{ClusterName: tt.clusterName})

            err := env.GetWorkflowError()
            if tt.wantErr && err == nil {
                t.Fatal("expected error, got nil")
            }
            if !tt.wantErr && err != nil {
                t.Fatalf("unexpected error: %v", err)
            }
        })
    }
}
```

**Split validation tests from execution tests.**

If your workflow has a constructor that validates its input, test that separately without any workflow environment:

```go
func TestNewDeployClusterWorkflow(t *testing.T) {
    tests := []struct {
        name      string
        input     *pb.DeployClusterInput
        wantErr   bool
        errSubstr string
    }{
        {name: "success", input: &pb.DeployClusterInput{ClusterName: "prod"}, wantErr: false},
        {name: "error - empty name", input: &pb.DeployClusterInput{}, wantErr: true, errSubstr: "cluster_name"},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            wf, err := newDeployClusterWorkflow(tt.input)
            if tt.wantErr {
                if err == nil { t.Fatal("expected error") }
                if tt.errSubstr != "" && !strings.Contains(err.Error(), tt.errSubstr) {
                    t.Errorf("want error containing %q, got: %v", tt.errSubstr, err)
                }
            } else {
                if err != nil { t.Fatalf("unexpected error: %v", err) }
            }
        })
    }
}
```

Constructor tests run in milliseconds — no Temporal environment needed.

### Why workflow tests matter for replay safety

Temporal replays workflows from event history. If new code executes activities in a different order than the original run, replay breaks and the workflow is stuck. Workflow unit tests catch this by validating that the same inputs always produce the same activity call sequence.

## Layer 2: Activity Tests

Activity tests mock the client interface. Test three layers in order:

1. **Validation errors** — caught before any client call is made
2. **Success path** — valid input, client returns success
3. **Client errors** — valid input, client returns each type of error

```go
// activities_test.go
package myservice

import (
    "errors"
    "testing"

    pb "github.com/myorg/myproject/gen/myservice/v1"
    mocks "github.com/myorg/myproject/mocks/myservice/v1"
    "github.com/stretchr/testify/mock"
    "go.temporal.io/sdk/testsuite"
)

var (
    errNotFound       = errors.New("not found")
    errAlreadyExists  = errors.New("already exists")
    errConnRefused    = errors.New("connection refused")
)

func TestCreateThingActivity(t *testing.T) {
    tests := []struct {
        name      string
        input     *pb.CreateThingInput
        clientErr error
        wantErr   bool
    }{
        // Layer 1: validation
        {name: "error - empty name", input: &pb.CreateThingInput{}, wantErr: true},

        // Layer 2: success
        {name: "success", input: &pb.CreateThingInput{Name: "my-thing"}, wantErr: false},

        // Layer 3: client errors
        {name: "error - already exists", input: &pb.CreateThingInput{Name: "my-thing"}, clientErr: errAlreadyExists, wantErr: true},
        {name: "error - connection refused", input: &pb.CreateThingInput{Name: "my-thing"}, clientErr: errConnRefused, wantErr: true},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            env := (&testsuite.WorkflowTestSuite{}).NewTestActivityEnvironment()
            mockClient := mocks.NewMockMyServiceClient(t)

            // Conditional mock: only expect a client call if validation would pass
            if tt.input.Name != "" {
                mockClient.EXPECT().
                    Create(mock.Anything, tt.input.Name).
                    Return("id-123", tt.clientErr)
            }

            acts := NewActivities(mockClient)
            env.RegisterActivity(acts.CreateThing)

            val, err := env.ExecuteActivity(acts.CreateThing, tt.input)

            if tt.wantErr {
                if err == nil { t.Fatal("expected error") }
                return
            }
            if err != nil {
                t.Fatalf("unexpected error: %v", err)
            }

            var out pb.CreateThingOutput
            if err := val.Get(&out); err != nil {
                t.Fatalf("get output: %v", err)
            }
            if out.Id == "" {
                t.Error("expected non-empty ID")
            }
        })
    }
}
```

**Conditional mocks** are the key pattern here. If the test case has an invalid input, the mock has no expectations set — mockery will fail the test if `Create` is called anyway, which confirms validation is working.

## Layer 3: Client Tests (Integration)

Client tests talk to a real service running in a Docker container. They use [testcontainers-go](https://golang.testcontainers.org) to spin up and tear down the service per test.

```go
//go:build integration
// +build integration

package myservice

import (
    "context"
    "errors"
    "fmt"
    "testing"
    "time"

    "github.com/testcontainers/testcontainers-go"
    "github.com/testcontainers/testcontainers-go/wait"
)

func TestMyServiceClient_RealOperations(t *testing.T) {
    if testing.Short() {
        t.Skip("skipping integration test in short mode")
    }

    ctx := context.Background()

    req := testcontainers.ContainerRequest{
        Image:        "myservice:latest",
        ExposedPorts: []string{"8080/tcp"},
        WaitingFor:   wait.ForHTTP("/health").WithPort("8080/tcp").WithStartupTimeout(30 * time.Second),
    }

    container, err := testcontainers.GenericContainer(ctx, testcontainers.GenericContainerRequest{
        ContainerRequest: req,
        Started:          true,
    })
    if err != nil {
        t.Fatalf("start container: %v", err)
    }
    defer container.Terminate(ctx)

    host, _ := container.Host(ctx)
    port, _ := container.MappedPort(ctx, "8080")
    client := NewMyServiceClient(fmt.Sprintf("http://%s:%s", host, port.Port()))

    t.Run("create and retrieve", func(t *testing.T) {
        id, err := client.Create(ctx, "my-thing")
        if err != nil {
            t.Fatalf("create: %v", err)
        }
        if id == "" {
            t.Error("expected non-empty ID")
        }
    })

    t.Run("not found returns ErrNotFound", func(t *testing.T) {
        _, err := client.Get(ctx, "does-not-exist")
        if !errors.Is(err, ErrNotFound) {
            t.Errorf("expected ErrNotFound, got: %v", err)
        }
    })
}
```

Key points:
- The `//go:build integration` tag keeps these out of normal `go test ./...` runs
- Start a fresh container per test function (not per subtest) for speed
- Always `defer container.Terminate(ctx)` — no cleanup on panic otherwise
- Assert against real client error types (`ErrNotFound`), not test sentinels — this verifies the client translates system error codes correctly

### Running client tests

```bash
# Unit tests only (fast, no Docker)
go test ./...

# Include integration tests (requires Docker)
go test -tags=integration ./...

# Skip integration tests explicitly
go test -short ./...
```

## Test File Layout

```
lib/<service>/<version>/
├── activities.go          # implementation
├── workflows.go           # implementation (if workflows exist)
├── client.go              # external system client
├── fx.go                  # dependency injection
├── workflows_test.go      # unit: mocks activities interface
├── activities_test.go     # unit: mocks client interface
└── client_test.go         # integration: real system, build tag required
```

## Key Principles

### Only mock your direct dependency

```go
// ✅ workflow mocks activities — its direct dependency
env.OnActivity("CreateThing", ...)

// ✅ activity mocks client — its direct dependency
mockClient.EXPECT().Create(...)

// ❌ never mock two layers deep
env.OnCall(httpClient.Do)    // workflow should not know about HTTP
```

### Conditional mocks match real code flow

If validation would reject the input before reaching the client, do not set expectations on the mock. Unexpected calls fail the test — which validates the validation layer.

```go
if tt.input.Name != "" {   // only expect call if validation passes
    mockClient.EXPECT().Create(...)
}
```

### Fresh setup per test case

```go
for _, tt := range tests {
    t.Run(tt.name, func(t *testing.T) {
        env := (&testsuite.WorkflowTestSuite{}).NewTestActivityEnvironment()
        mockClient := mocks.NewMockMyServiceClient(t)
        // ...
    })
}
```

Never share `env` or mock state across table-driven test cases.

### Test names are documentation

```go
// ✅ intent is obvious without reading the test body
{name: "success - creates role with multiple policies"},
{name: "error - empty role name"},
{name: "error - client connection refused"},

// ❌ useless
{name: "test1"},
{name: "error case"},
```

## Implementation Checklist

### Workflow tests
- [ ] Constructor tests: each required field, each format constraint, no env needed
- [ ] Execution tests: happy path, each error path, each branch
- [ ] Fresh `WorkflowTestSuite` per test case
- [ ] Mock only the activities/child workflows — nothing deeper
- [ ] Descriptive test names showing the flow

### Activity tests
- [ ] Sentinel errors defined at package level
- [ ] Validation cases first (before happy path)
- [ ] Happy path (minimal valid input)
- [ ] Each client error type separately
- [ ] Conditional mocks that mirror code flow
- [ ] Verify output shape for success cases

### Client tests
- [ ] `//go:build integration` tag
- [ ] `testing.Short()` skip guard
- [ ] Proper `WaitingFor` health check on container
- [ ] `defer container.Terminate(ctx)`
- [ ] Real error types checked, not sentinels
